# VOC 자동 분석 파이프라인

## Executive Summary

SaaS 제품 팀이 주간 수백 건의 고객 피드백을 처리하는 방식은 오랫동안 변하지 않았다 — 담당자가 직접 읽고, 분류하고, 회의에서 논의하는 과정. 이 수작업 사이클은 피로를 누적시킬 뿐 아니라, 기업고객의 이탈 예고 메시지가 일반 문의 티켓 사이에 묻혀 48시간이 지나도록 발견되지 않는 구조적 실패를 만들어냈다. 이 파이프라인은 `VOCPipeline.run()` 한 줄로 분류-점수화-리포팅의 전 과정을 자동화하고, 이탈 위험 신호를 실시간으로 선별하여 CS 팀이 즉시 대응할 수 있는 체계로 전환한다.

핵심은 **'Frequency × Impact Score'** 라는 독자적 우선순위 로직이다. 단순 빈도 집계는 가장 많이 언급된 것을 올리고, 단순 심각도 기준은 가장 급한 것만 올린다 — 두 가지 모두 SaaS 서비스에서는 불완전하다. 이 프레임워크는 고객 등급(기업고객 ×3 / 유료 ×2 / 무료 ×1), 심각도(×1~3), 감정 온도(부정 ×1.5), 유형(버그 ×1.3)을 곱하여 행 단위 Impact Score를 산정하고, 동일 유형의 발생 빈도(Frequency)를 다시 곱함으로써 "지금 팀이 집중해야 할 교차점"을 수치로 드러낸다. 이탈 의사를 표현한 피드백(해지·분쟁·탈퇴)에는 별도 ×2 하드 가중치를 적용해, 단 한 건이라도 즉시 최상위로 부상하도록 설계했다.

이 시스템의 최종 목표는 분석 정확도가 아니라 비즈니스 결과다. Impact Score 상위 이슈 24시간 내 선제 대응은 Chargeback 리스크와 계약 해지를 조기 차단하여 MRR 방어로 직결되고, 기업고객 기능 요청을 Priority Score로 정량화하는 로드맵 반영 프로세스는 유료 전환율과 LTV 증대의 근거가 된다. 매주 월요일 자동 발행되는 주간 리포트는 CS·개발·기획 팀이 같은 데이터에서 각자의 행동 지침을 추출하는 단일 소스로 기능하며, 조직 전체가 고객 신호에 응답하는 속도를 구조적으로 단축한다.

---


> 반복되는 수작업 VOC 분류를 자동화하고, 비즈니스 임팩트 기반으로 즉시 대응이 필요한 이슈를 정확히 식별하는 파이프라인

## 1. 프로젝트 개요 — 해결한 고통

| 기존 고통 | 이 프로젝트의 해결 |
|---|---|
| 수백 건 VOC를 수작업 분류 → 담당자 피로 누적 | `VOCPipeline.run()` 1행으로 전체 자동화 |
| 경계 케이스(버그 vs 기능요청)의 일관성 없는 분류 | DEC-01~DEC-06 분류 원칙 문서화, 전략 패턴으로 일관 적용 |
| 긴급 이슈가 일반 티켓에 묻혀 뒤늦게 발견 | 비즈니스 임팩트 점수(priority_score) 기반 즉시대응 TOP N 자동 선별 |
| 채널·분류 기준 변경 시 코드 전면 수정 필요 | `PipelineConfig` 단일 객체만 수정 — 코드 무수정 |

**분석 대상**: `cleaned_voc.csv` · 300건 · 2026-05-04 ~ 2026-06-30

## 2. 기술적 파이프라인

```
*.csv 입력
    └─▶ [1] Load       인코딩 자동 감지 (UTF-8 / UTF-8-BOM / CP949)
    └─▶ [2] Validate   필수 컬럼 검증, null 제거
    └─▶ [3] Classify   규칙 기반 or LLM (전략 패턴 — 런타임 주입)
    └─▶ [4] Triage     severity=높음 & 감정=부정 필터 → user_type 랭크 정렬 → TOP-N
    └─▶ [5] Report     classified_voc.csv + report.json 자동 출력
```

**핵심 설계 원칙**

- **`PipelineConfig`** — 분류 키워드·가중치·임계값을 코드 외부로 완전 분리
- **Strategy Pattern** — `llm_classifier` 함수 주입만으로 규칙 기반 ↔ LLM API 전환
- **인코딩 자동 감지** — 다국어 환경에서도 파일 깨짐 없이 로드

```python
# 단 3줄로 파이프라인 실행
cfg    = PipelineConfig(top_n=3)
result = VOCPipeline(cfg).run("cleaned_voc.csv", "./output")
print(f"즉시대응 TOP-N: {len(result.top_n)}건")
```

## 3. 핵심 전략

### 3-1. 유형 분류 현황 (유효 분류 297건)

> 원본 300건 수신 → content 기준 중복 3건(F012=F300, F016=F069, F186=F221) 제거 → 유효 분류 **297건**

| 유형 | 건수 | 비율 | 주요 패턴 |
|---|---|---|---|
| 일반문의 | 156 | 52.5% | 사용법·정책·결제 관련 문의 |
| 기능요청 | 83 | 27.9% | 알림·캘린더·연동 기능 집중 |
| 버그 | 33 | 11.1% | 인증 장애·데이터 소실 포함 |
| 칭찬 | 25 | 8.4% | 협업·간트차트·CS 응대 호평 |

### 3-2. 우선순위 엔진

```
TOP-N 선정 기준 (3단계 필터+정렬)

  [1단계] severity = 높음  AND  감정 = 부정  →  후보군 필터링
  [2단계] user_type 우선순위 정렬: 기업고객(0) > 유료(1) > 무료(2) > N/A(3)
  [3단계] 동점 시 date 내림차순 (최신 건 우선)

  → 후보군 14건 중 상위 TOP 3 선정 (DEC-05: 유형분류와 독립 설계)
```

### 3-3. 즉시대응 TOP 3

| 순위 | ID | 핵심 이슈 | 선별 이유 |
|---|---|---|---|
| 🔴 1위 | F046 | 팀 전체 3일 로그인 불가 + 해지 위협 | 기업고객, 업무 마비, 이탈 의사 표현 |
| 🟠 2위 | F026 | 업데이트 후 문서 수십 건·계약서 소실 | 기업고객, 법적 리스크, 복구 불가 |
| 🟡 3위 | F170 | 2단계 인증 SMS 미수신·로그인 불가 | 유료고객, 인증 서비스 장애 |

### 3-4. 임팩트 최우선 개선 기획안

> **우선순위 판단 기준**: 빈도(동일 유형 발생 건수) × 임팩트(고객등급 × 심각도 × 감정) × 긴급도(이탈 위협 신호 ×2). 세 축의 교점이 가장 높은 클러스터를 P0으로 지정합니다. (상세 로직: `decisions.md` R-03~R-05)

#### 🔴 P0 — 인증 파이프라인 안정화 (즉시 착수)

**선정 근거**

| 기준 | 값 | 판단 |
|---|---|---|
| **빈도** | 버그 33건 중 인증 장애 2건 (TOP3의 2/3 점유) | 단일 장애 유형 최다 |
| **임팩트** | 기업고객 × 심각도 높음 × 감정 부정 = 최고 등급 | Priority Score 최상위 |
| **긴급도** | F046 "해지하겠습니다" — 이탈 위협 신호 감지 ×2 | 이탈 직결, 즉시 대응 |

**원문 근거**: *"로그인 자체가 막혀서 팀 전체가 사흘째 업무를 못 하고 있습니다. 해지하겠습니다"* (F046) / *"SMS가 도착을 안 해서 로그인 자체가 막혔어요"* (F170)

| 해결 과제 | 기대 효과 | 담당 |
|---|---|---|
| 2FA SMS 실패 시 이메일·앱 인증 폴백 추가 | 인증 실패율 → 0% | 개발 |
| 로그인 세션 토큰 갱신 주기 최적화 | 비정상 세션 만료 제거 | 개발 |
| 인증 서비스 실시간 헬스체크 API | 장애 조기 감지, MTTR 단축 | 인프라 |
| 기업고객 긴급 에스컬레이션 채널 신설 | 이탈 위협 건 4h 내 대응 | CS |

> * **기대 효과**: 현 평균 대응시간 72h → 목표 4h 이내 대응 (기업 고객 SLA 준수 및 MRR 방어)

## 4. 향후 확장성

| 시나리오 | 변경 범위 | 구체적 방법 |
|---|---|---|
| 신규 채널 추가 (카카오·이메일 등) | `PipelineConfig`만 | `scoring_weights["channel"]` 항목 추가 |
| 분류 기준 변경 (유형 추가·삭제) | `PipelineConfig`만 | `bug_keywords` 등 리스트 수정 |
| LLM 분류기 교체 (Claude API 등) | 함수 주입 1줄 | `PipelineConfig(llm_classifier=my_fn)` |
| 우선순위 기준 재조정 | `PipelineConfig`만 | `user_priority` dict 값 변경 |

> **설계 원칙**: 비즈니스 로직(가중치·키워드)과 파이프라인 코드를 분리했기 때문에
> 분류 기준이 바뀌어도 `voc_pipeline.py`는 수정하지 않는다.
